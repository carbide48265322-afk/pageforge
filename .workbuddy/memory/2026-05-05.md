## SSE 架构重构（2026-05-05 完成）

### 设计决策
将所有自定义 SSE 事件的类型从 `event:` 字段移到 `data.type` 内部，统一用 `event: message`。
前端只需监听一个 `message` 事件，通过 `data.type` switch 分发。

### 改动文件
1. **`backend/app/graph/event_emitter.py`**: `emit_event()` 固定输出 `event: "message"`，原 event 参数值放入 `data.type`
2. **`backend/app/api/messages.py`**: `on_custom_event` 处理兼容新旧格式，新格式统一输出 `event: message`
3. **`frontend/src/services/sse/SseEventDispatcher.ts`**: GET 模式只监听 `message` 事件，POST 模式从 `data.type` 分发

### 兼容性
- HANDLER_MAP key 不变（还是 `thinking_delta` 等）
- handlers 收到的 data 多了 `type` 字段，但解构不影响
- `useSSEv2.ts` 订阅回调用 `as` 类型断言，不受影响
- `messages.py` 保留旧格式兼容分支

### 验证
- Python `ast.parse` ✅
- TypeScript `tsc --no Exit` ✅
- commit: `071a44c` on `feature/v2-node-implementation`

## Handler 层删除（2026-05-05 完成）

### 发现
Handler 层完全多余：
1. 后端每条 SSE data 已带完整数据（非增量），handler 的累积逻辑是重复工作
2. `handleSSEEvent()` 调用 handler 但不接返回值，handler 产出被丢弃
3. `useSSEv2` 的 subscriber 直接用原始 `data.content`，不依赖 handler

### 删除内容
- `frontend/src/services/sse/handlers/` 目录（9 个文件，~800 行）
- `SseEventDispatcher.ts` 中的 HANDLER_MAP 和 9 个 handler import
- `handleSSEEvent()` 中 handler 路由逻辑，只保留 subscriber 通知
- `index.ts` 中的 handler re-export

### 结果
- 架构从三层变两层：Dispatcher（分发）→ useSSEv2（聚合+转换）→ ChatPanelV2（渲染）
- TypeScript 编译 ✅
